Bulletin Upload Benchmarks • 19 March 2026

Uploads work through 500MB. 1GB breaks.

Small and medium files upload fine. 500MB completes but needs retries. At 1GB the problem is not speed. The chain storage path loses continuity partway through, and the upload fails.

Data: benchmark profiles, consolidated JSON outputs, rerun logs, 1GB resume manifests.

500MB works. 1GB does not. The bottleneck is not the client or the network. It is the chain storage path losing continuity under sustained load.

Upload Lifecycle

Step 1
Create test file
Deterministic file generated on disk. Resumable upload opened.
Step 2
Chunk into blocks
File split into 2MB blocks. Each block gets a CID.
Step 3
Store via Bulletin
Chunks submitted sequentially. Single-wave baseline.
Step 4
Pin and finalise
This is where it breaks. At 1GB: not pinned, stop-call fails.
Step 5
CID resolves
If the chain path holds, the root CID becomes verifiable.
Chunking works fine. The problem is downstream: the chain cannot sustain the store-then-pin sequence over hundreds of consecutive blocks.

Degradation By Size

Stable
Clean uploads. No retries needed. Everything works as expected. Predictable behaviour.
50MB – 100MB
Slowing
Time grows fast. Memory climbs. Throughput drops. Still completes, but the system is clearly under more pressure.
500MB
Fragile
Success depends on retries. It finishes, but only because the retry logic catches failures along the way.
1GB
Broken
Chain continuity collapses. not pinned, stop-call failure, WebSocket halt, 3x SIGKILL.
The 1GB failure is qualitatively different. It is not "slow." The storage path stops working partway through.

Where It Breaks

Chain storage continuity

  • 1GB failed with not pinned (stop-call).
  • Same attempt logged WS halt (3).
  • This was a single-wave run. No aggressive parallelism.
  • So the chain path itself is the suspect, not concurrency.

Memory pressure

  • Peak RSS: 1.48 GB at 50MB, 1.61 GB at 100MB, 1.67 GB at 500MB.
  • Three SIGKILL events during the 1GB run.
  • Memory does not explain the chain error, but it reduces margin.
The main bottleneck is Bulletin's ability to sustain sequential storage at scale. Memory pressure is a contributing factor, not the root cause.

Degradation Within The Last Attempt

What the log shows

  • Wave 86: 2.61 MB/s, ETA 5m26s.
  • Wave 93: 1.13 MB/s, ETA 12m22s.
  • Throughput is falling while progress continues.

What that means

  • The system is not slow-and-steady. It is degrading.
  • The delivery path gets less stable as the run ages.
  • Resume helps, but cannot make 1GB reliable on its own.

What To Do Next

For users now

  • Uploads through 500MB work. Use them.
  • 1GB is experimental. Expect failures.
  • Keep resume enabled. It preserves partial progress.

For the team

  • Instrument stop-call handling and pin persistence.
  • Measure WebSocket halt frequency and finalisation latency.
  • Do not increase concurrency until serial 1GB is stable.
The goal is not "make 1GB work once." It is "make 1GB boring."
